Automate AWS Marketplace Purchase Notifications for Procurement Teams Using Amazon EventBridge and AWS Lambda

by Matthew Brend | on 13 JUL 2026 | in AWS Marketplace Amazon EventBridge AWS Lambda Cloud Financial Management Enterprise

In large enterprises, the teams purchasing software and the teams paying for it are often completely disconnected. Engineering subscribes to a Marketplace product on a Tuesday afternoon; procurement doesn't find out until an invoice lands in their queue weeks later. By then, there's no PO on file, nobody's sure which cost center it belongs to, and the clock is ticking toward a late payment fee.

This post walks through building an automated notification pipeline that closes that gap. When any team in your organization subscribes to an AWS Marketplace product, your procurement team gets an immediate email with everything they need to start the PO process — product name, estimated cost, pricing terms, who initiated the purchase, and the PO number if one was provided at checkout. The solution works across multiple accounts in an AWS Organization using cross-account EventBridge event forwarding, and deploys in under an hour.

Background: AWS Marketplace emits agreement lifecycle events to Amazon EventBridge for all parties involved in a transaction — manufacturers (ISVs), channel partners, and buyers. Each party receives events relevant to their role. This post focuses exclusively on the buyer-side events and how to build an enrichment pipeline on top of them. For a comprehensive overview of all Marketplace EventBridge events, see Enhance Agreement Notifications in AWS Marketplace with Amazon EventBridge.

Solution overview

AWS Marketplace emits agreement lifecycle events to Amazon EventBridge whenever a subscription is created, modified, or approaching expiration. We'll capture these events, enrich them with details from the Marketplace Agreement and Discovery APIs, and deliver a procurement-ready notification via Amazon SNS.

If you just need a basic "a purchase happened" alert without the enriched details, skip ahead to the simplified alternative — it's three CLI commands and zero code. The full pipeline below is for teams that need product names, dollar amounts, and purchaser identity delivered directly to procurement without console access.

Architecture

How it works

  1. A user subscribes to a product in AWS Marketplace from any linked account.
  2. AWS Marketplace emits a Purchase Agreement Created - Acceptor event to the default EventBridge bus in that account.
  3. A lightweight EventBridge rule forwards the event cross-account to the hub account's default event bus.
  4. In the hub account, an EventBridge rule triggers a Lambda function.
  5. The Lambda assumes a cross-account role in the source account and calls Marketplace APIs to enrich the event with product name, pricing, PO number, and purchaser identity.
  6. The enriched notification is published to an SNS topic, which delivers it to your procurement team via email, Slack, or a ticketing system like ServiceNow.

Events captured

EventWhen it firesUse case
Purchase Agreement Created - AcceptorNew purchase, renewal, or replacementStart PO process
Purchase Agreement Amended - AcceptorTerms or pricing modifiedUpdate existing PO
Purchase Agreement Ending - Acceptor90, 60, and 30 days before expirationPlan renewal PO

Prerequisites

  • An AWS Organization with multiple accounts
  • A designated hub account for centralized notification processing
  • AWS CLI v2 or access to the AWS Management Console
  • Python 3.12 Lambda runtime

Step 1: Create the SNS topic (hub account)

Create an SNS topic in your hub account and subscribe your procurement team's email or notification endpoint.

# Create the SNS topic
aws sns create-topic --name MarketplacePurchaseNotifications --region us-east-1

# Subscribe your procurement team
aws sns subscribe \
  --topic-arn "arn:aws:sns:us-east-1:HUB_ACCOUNT_ID:MarketplacePurchaseNotifications" \
  --protocol email \
  --notification-endpoint "procurement-team@yourcompany.com" \
  --region us-east-1
Important: Confirm the subscription by clicking the link in the confirmation email sent to the procurement team's address.

Step 2: Create the enrichment Lambda (hub account)

IAM role for the Lambda

Create an execution role that allows the Lambda to read Marketplace agreements, publish to SNS, and assume roles in linked accounts:

Lambda function code

The enrichment Lambda receives the EventBridge event, assumes a role in the source account, queries multiple Marketplace APIs, and publishes a formatted notification. Here's the core logic:

import json, os, logging, boto3
from datetime import datetime, timedelta, timezone

sns_client = boto3.client('sns', region_name='us-east-1')
sts_client = boto3.client('sts', region_name='us-east-1')

def lambda_handler(event, context):
    detail = event.get('detail', {})
    agreement_id = detail['agreement']['id']
    source_account = event['account']

    # Assume role in the account where the purchase was made
    clients = get_cross_account_clients(source_account)

    # Enrich with product name, pricing, PO number, and purchaser identity
    agreement = clients['agreement'].describe_agreement(agreementId=agreement_id)
    product = clients['discovery'].get_product(
        productId=agreement['proposalSummary']['resources'][0]['id']
    )
    terms = clients['agreement'].get_agreement_terms(agreementId=agreement_id)

    # Build and send notification
    notification = build_notification(event, agreement, product, terms)
    sns_client.publish(
        TopicArn=os.environ['SNS_TOPIC_ARN'],
        Subject=f"[Marketplace PO] {product['productName']}",
        Message=notification
    )
Full source code: The complete Lambda function (~250 lines) with error handling, CloudTrail lookup for purchaser identity, PO number retrieval, and dynamic subject line logic is available in the companion GitHub repository.

Step 3: Create the EventBridge rule (hub account)

aws events put-rule \
  --name MarketplacePurchaseEnricher-Trigger \
  --event-pattern '{
    "source": ["aws.agreement-marketplace"],
    "detail-type": [
      "Purchase Agreement Created - Acceptor",
      "Purchase Agreement Amended - Acceptor",
      "Purchase Agreement Ending - Acceptor"
    ]
  }' \
  --state ENABLED --region us-east-1

# Grant EventBridge permission to invoke Lambda
aws lambda add-permission \
  --function-name MarketplacePurchaseEnricher \
  --statement-id EventBridgeInvoke \
  --action lambda:InvokeFunction \
  --principal events.amazonaws.com \
  --source-arn arn:aws:events:us-east-1:HUB_ACCOUNT_ID:rule/MarketplacePurchaseEnricher-Trigger

# Add the Lambda as target
aws events put-targets \
  --rule MarketplacePurchaseEnricher-Trigger \
  --targets '[{"Id":"EnricherLambda","Arn":"arn:aws:lambda:us-east-1:HUB_ACCOUNT_ID:function:MarketplacePurchaseEnricher"}]'

Step 4: Enable cross-account event forwarding (hub account)

Understanding event timing: The Purchase Agreement Created event fires when a buyer accepts an offer — which can occur before the agreement's start date for future-dated agreements. This means your procurement team may receive a notification days or weeks before the software is actually activated or billed. This is ideal for PO workflows, as it gives finance lead time to process the purchase order before charges begin.
Auto-renewal tip: The Purchase Agreement Ending event includes an autoRenewalEnabled field. If auto-renewal is ON, your Lambda can adjust the notification subject to indicate that renewal will happen automatically — the procurement team only needs to act if they want to cancel the renewal rather than create a new PO.

Allow linked accounts to forward events to the hub account's default event bus. For organization-wide access:

aws events put-permission \
  --action events:PutEvents \
  --principal "*" \
  --statement-id "AllowOrganization" \
  --condition '{"Type":"StringEquals","Key":"aws:PrincipalOrgID","Value":"o-your-org-id"}' \
  --region us-east-1

Step 5: Deploy linked account resources (CloudFormation StackSet)

Each account that makes Marketplace purchases needs a forwarding rule and a cross-account IAM role. You probably don't need this in every account — most organizations have a subset of accounts (10–15 out of hundreds) where Marketplace subscriptions actually happen. Check your AWS Cost Explorer with a BILLING_ENTITY = "AWS Marketplace" filter grouped by LINKED_ACCOUNT to identify which accounts have Marketplace spend.

Deploy to those accounts using a CloudFormation StackSet with explicit account targeting (not the full organizational unit):

AWSTemplateFormatVersion: '2010-09-09'
Description: Marketplace Purchase Notification - Linked Account Resources

Parameters:
  HubAccountId:
    Type: String
    Description: Account ID where the enrichment Lambda runs

Resources:
  CrossAccountRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: MarketplacePurchaseEnricher-CrossAccount
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              AWS: !Sub "arn:aws:iam::${HubAccountId}:role/MarketplacePurchaseEnricher-LambdaRole"
            Action: sts:AssumeRole
      Policies:
        - PolicyName: MarketplaceEnricherReadPolicy
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action:
                  - aws-marketplace:DescribeAgreement
                  - aws-marketplace:GetAgreementTerms
                  - aws-marketplace:ListAgreementCharges
                  - aws-marketplace:GetProduct
                  - aws-marketplace-discovery:GetProduct
                  - cloudtrail:LookupEvents
                Resource: "*"

  EventForwarderRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: MarketplaceEventForwarder-Role
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              Service: events.amazonaws.com
            Action: sts:AssumeRole
      Policies:
        - PolicyName: ForwardToHubEventBus
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action: events:PutEvents
                Resource: !Sub "arn:aws:events:us-east-1:${HubAccountId}:event-bus/default"

  ForwardingRule:
    Type: AWS::Events::Rule
    Properties:
      Name: MarketplaceEvent-ForwardToHub
      EventPattern:
        source: ["aws.agreement-marketplace"]
        detail-type:
          - "Purchase Agreement Created - Acceptor"
          - "Purchase Agreement Amended - Acceptor"
          - "Purchase Agreement Ending - Acceptor"
      Targets:
        - Id: HubAccountEventBus
          Arn: !Sub "arn:aws:events:us-east-1:${HubAccountId}:event-bus/default"
          RoleArn: !GetAtt EventForwarderRole.Arn
Scoping the deployment: When creating the StackSet, use the Deploy to accounts option and specify only the account IDs with Marketplace activity — not Deploy to organizational units. This avoids deploying unused roles and rules to sandbox, dev, and shared-services accounts that will never make Marketplace purchases. You can always add accounts later as purchasing patterns evolve.

Key design decisions

Why cross-account role assumption? Marketplace Agreement APIs must be called from the account that owns the agreement. The EventBridge event arrives in the hub account, but the data lives in the source account. STS AssumeRole bridges this gap with a single Lambda deployment.

Why Marketplace Discovery API for product names? The Marketplace Catalog API (DescribeEntity) is a seller-side API — buyers get AccessDenied. We burned some time on this before discovering that the Marketplace Discovery API (GetProduct) is the buyer-side equivalent. It returns the product title, vendor name, and even the product category — exactly what a procurement team needs to identify what was purchased without logging into the console.

Why EventBridge forwarding instead of deploying Lambda in every account? Centralized logic means one place to update notification format, add enrichment sources, or change delivery targets. The per-account forwarding rule is stateless and trivial — ideal for StackSet deployment.

Why capture the "Ending" event? It fires at 90, 60, and 30 days before expiration — giving procurement teams lead time to budget and prepare renewal POs before the agreement lapses.

Simplified alternative: No-code notifications

If your procurement team just needs to know that a purchase happened — and can look up the details themselves in the Marketplace console — you can skip the Lambda enrichment entirely. The two simplest approaches:

Option A: AWS User Notifications (console, zero code)

  1. Navigate to AWS User Notifications in the console.
  2. Create a notification configuration for source aws.agreement-marketplace.
  3. Select Purchase Agreement Created - Acceptor as the event type.
  4. Add your procurement team's email, Slack workspace, or Microsoft Teams channel as the delivery target.

That's it. No IAM roles, no Lambda, no EventBridge rules to manage. The notification will include the agreement ID, account, and event type — but not the product name, dollar amount, or who made the purchase.

Option B: EventBridge → SNS direct (three CLI commands)

# 1. Create SNS topic and subscribe procurement email
aws sns create-topic --name MarketplacePurchaseAlerts --region us-east-1
aws sns subscribe \
  --topic-arn "arn:aws:sns:us-east-1:ACCOUNT_ID:MarketplacePurchaseAlerts" \
  --protocol email \
  --notification-endpoint "procurement@yourcompany.com" \
  --region us-east-1

# 2. Create EventBridge rule
aws events put-rule \
  --name MarketplacePurchaseAlert \
  --event-pattern '{"source":["aws.agreement-marketplace"],"detail-type":["Purchase Agreement Created - Acceptor"]}' \
  --region us-east-1

# 3. Target the SNS topic
aws events put-targets \
  --rule MarketplacePurchaseAlert \
  --targets '[{"Id":"SNS","Arn":"arn:aws:sns:us-east-1:ACCOUNT_ID:MarketplacePurchaseAlerts"}]' \
  --region us-east-1

This gives you a raw JSON event in the email — not pretty, but functional. The procurement team gets the agreement ID and a link to look it up. For many organizations, this is enough to trigger their PO workflow.

When to use the full enrichment pipeline

Invest in the Lambda enrichment approach (described in this post) when:

  • Your procurement team doesn't have AWS console access
  • Purchases happen across many accounts and you need cross-account visibility
  • You need the dollar amount, product name, and purchaser identity in the notification itself
  • You're integrating with a ticketing system (ServiceNow, Jira) that needs structured fields
  • Late fees from delayed PO processing are costing you money

Extending the solution

  • Slack integration: Replace the SNS email subscription with a Lambda that formats a Slack message using Block Kit.
  • ServiceNow integration: Subscribe an HTTPS endpoint that creates incidents automatically.
  • Approval workflows: Add a Step Functions workflow requiring manager approval before high-value agreements activate.
  • Cost thresholds: Notify only on purchases above a dollar threshold, or escalate high-value purchases to leadership.

Conclusion

By combining Amazon EventBridge's native Marketplace integration with a lightweight Lambda enrichment function and cross-account role assumption, you can give your procurement team real-time visibility into AWS Marketplace purchases — complete with product names, pricing, PO numbers, and purchaser identity.

This eliminates the gap between purchase and PO creation, reduces invoice processing delays, and gives finance teams the lead time they need to manage cloud software spend effectively. The entire solution deploys in under an hour, costs virtually nothing to run, and scales automatically across your entire AWS Organization.
